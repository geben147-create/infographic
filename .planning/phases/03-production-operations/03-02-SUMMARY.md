# Phase 03 Plan 02: Batch Runner, Schedule CLI, and Cost Column Summary

**One-liner:** CLI batch runner and Temporal schedule creator with per-video cost persisted in SQLite via Alembic migration.

## What Was Built

### Task 1: Batch Runner CLI + Schedule CLI (TDD)

**scripts/batch_runner.py** — Sequential batch pipeline executor:
- Reads a JSON file with `[{topic, channel_id, quality_gate_enabled?}]` items
- Calls `client.start_workflow(ContentPipelineWorkflow.run, ...)` for each item
- Awaits `handle.result()` before starting next workflow (sequential, not parallel — required for single-GPU constraint)
- Handles workflow failures gracefully: logs error and continues to next item
- Accepts injectable `client` parameter for unit testing without live Temporal

**scripts/schedule_video.py** — Temporal one-time schedule creator:
- Creates a `Schedule` with `ScheduleCalendarSpec` wrapping each datetime component in `ScheduleRange(start=N)` (not bare ints — Temporal SDK requirement)
- Sets `ScheduleState(limited_actions=True, remaining_actions=1)` to ensure single-fire execution
- Returns `schedule_id` string for caller tracking
- Accepts injectable `client` parameter for unit testing

**tests/test_batch_runner.py** — 4 tests:
1. `run_batch()` calls `start_workflow` once per item
2. Sequential execution order: start → result → start → result
3. Workflow failure handling: failed item logged, next item processed
4. Correct PipelineParams (topic, channel_id) passed to start_workflow

**tests/test_schedule_video.py** — 4 tests:
1. `schedule_one()` calls `create_schedule` exactly once
2. ScheduleCalendarSpec fields use `ScheduleRange` wrappers (not bare ints)
3. `ScheduleState.limited_actions=True` and `remaining_actions=1` enforced
4. Returned `schedule_id` matches first arg passed to `create_schedule`
